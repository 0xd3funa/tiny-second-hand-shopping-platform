# Tiny Second-hand Shopping Platform 요구사항 분석

## 1. 프로젝트 개요

### 1.1 프로젝트명

Tiny Second-hand Shopping Platform

### 1.2 개발 목적

사용자가 중고 상품을 등록하고 검색하며 다른 사용자와 소통할 수 있는
소규모 중고거래 플랫폼을 개발한다.

기능 구현뿐만 아니라 소프트웨어 개발 전 과정에서 보안 약점을 확인하고
시큐어코딩 원칙을 적용하는 것을 목표로 한다.

### 1.3 프로젝트 범위

본 프로젝트에서는 다음 기능을 제공한다.

- 회원가입 및 로그인
- 사용자 프로필 관리
- 중고 상품 등록 및 관리
- 상품 목록, 상세 조회 및 검색
- 전체 채팅 및 사용자 간 1:1 채팅
- 사용자 및 상품 신고
- 신고 횟수에 따른 자동 차단
- 사용자 간 가상 잔액 송금
- 관리자 페이지

실제 금융기관이나 결제 서비스와 연결하지 않고,
교육용 가상 잔액을 이용하여 송금 기능을 구현한다.

## 2. 사용자 유형

### 2.1 비회원

- 회원가입을 할 수 있다.
- 로그인할 수 있다.
- 차단되지 않은 상품 목록을 조회할 수 있다.
- 상품 이름으로 상품을 검색할 수 있다.
- 상품 상세 정보를 조회할 수 있다.

### 2.2 일반 사용자

비회원의 기능을 포함하며 다음 기능을 추가로 사용할 수 있다.

- 자신의 프로필을 확인할 수 있다.
- 소개글을 수정할 수 있다.
- 비밀번호를 변경할 수 있다.
- 상품을 등록할 수 있다.
- 자신이 등록한 상품을 수정하거나 삭제할 수 있다.
- 상품의 판매 상태를 변경할 수 있다.
- 전체 채팅에 참여할 수 있다.
- 다른 사용자와 1:1 채팅을 할 수 있다.
- 악성 사용자 또는 상품을 신고할 수 있다.
- 다른 사용자에게 가상 잔액을 송금할 수 있다.
- 자신의 송금 내역을 확인할 수 있다.

### 2.3 관리자

- 전체 사용자를 조회하고 관리할 수 있다.
- 사용자의 계정 상태를 변경할 수 있다.
- 전체 상품을 조회, 차단 또는 삭제할 수 있다.
- 모든 신고 내역을 확인할 수 있다.
- 부적절한 채팅 메시지를 삭제할 수 있다.
- 전체 송금 내역을 조회할 수 있다.

## 3. 기능 요구사항

### 3.1 사용자 관리

- FR-USER-01: 사용자는 고유한 아이디로 회원가입할 수 있어야 한다.
- FR-USER-02: 중복된 아이디로 회원가입할 수 없어야 한다.
- FR-USER-03: 사용자는 아이디와 비밀번호로 로그인할 수 있어야 한다.
- FR-USER-04: 사용자는 로그아웃할 수 있어야 한다.
- FR-USER-05: 사용자는 자신의 프로필과 소개글을 확인할 수 있어야 한다.
- FR-USER-06: 사용자는 자신의 소개글을 수정할 수 있어야 한다.
- FR-USER-07: 사용자는 현재 비밀번호를 확인한 후 비밀번호를 변경할 수 있어야 한다.
- FR-USER-08: 휴면 처리된 사용자는 로그인하거나 주요 기능을 사용할 수 없어야 한다.

### 3.2 상품 관리

- FR-PRODUCT-01: 로그인 사용자는 상품을 등록할 수 있어야 한다.
- FR-PRODUCT-02: 상품에는 상품명, 설명, 가격 및 사진이 포함되어야 한다.
- FR-PRODUCT-03: 사용자는 차단되지 않은 전체 상품을 조회할 수 있어야 한다.
- FR-PRODUCT-04: 사용자는 상품 상세 정보를 조회할 수 있어야 한다.
- FR-PRODUCT-05: 판매자는 자신이 등록한 상품을 수정할 수 있어야 한다.
- FR-PRODUCT-06: 판매자는 자신이 등록한 상품을 삭제할 수 있어야 한다.
- FR-PRODUCT-07: 판매자는 상품 상태를 판매 중, 예약 중 또는 판매 완료로 변경할 수 있어야 한다.
- FR-PRODUCT-08: 사용자는 상품명으로 상품을 검색할 수 있어야 한다.
- FR-PRODUCT-09: 차단된 상품은 일반 상품 목록에 표시되지 않아야 한다.

### 3.3 채팅

- FR-CHAT-01: 로그인 사용자는 전체 채팅에 참여할 수 있어야 한다.
- FR-CHAT-02: 로그인 사용자는 다른 사용자와 1:1 채팅을 할 수 있어야 한다.
- FR-CHAT-03: 채팅 메시지는 작성자와 작성 시간을 표시해야 한다.
- FR-CHAT-04: 채팅 메시지는 데이터베이스에 저장되어야 한다.

### 3.4 신고 및 차단

- FR-REPORT-01: 로그인 사용자는 부적절한 상품을 신고할 수 있어야 한다.
- FR-REPORT-02: 로그인 사용자는 부적절한 사용자를 신고할 수 있어야 한다.
- FR-REPORT-03: 신고자는 신고 사유를 작성해야 한다.
- FR-REPORT-04: 동일 사용자가 같은 대상을 중복 신고할 수 없어야 한다.
- FR-REPORT-05: 서로 다른 사용자에게 3회 이상 신고된 상품은 자동으로 차단되어야 한다.
- FR-REPORT-06: 서로 다른 사용자에게 3회 이상 신고된 사용자는 자동으로 휴면 처리되어야 한다.
- FR-REPORT-07: 사용자는 자신이나 자신의 상품을 신고할 수 없어야 한다.

### 3.5 송금

- FR-TRANSFER-01: 로그인 사용자는 다른 사용자에게 가상 잔액을 송금할 수 있어야 한다.
- FR-TRANSFER-02: 보유 잔액을 초과하는 금액은 송금할 수 없어야 한다.
- FR-TRANSFER-03: 0원 이하의 금액은 송금할 수 없어야 한다.
- FR-TRANSFER-04: 사용자는 자기 자신에게 송금할 수 없어야 한다.
- FR-TRANSFER-05: 출금과 입금은 하나의 트랜잭션으로 처리되어야 한다.
- FR-TRANSFER-06: 송금 결과와 송금 내역이 데이터베이스에 기록되어야 한다.

### 3.6 관리자

- FR-ADMIN-01: 관리자는 전체 사용자를 조회할 수 있어야 한다.
- FR-ADMIN-02: 관리자는 사용자를 활성 또는 휴면 상태로 변경할 수 있어야 한다.
- FR-ADMIN-03: 관리자는 모든 상품을 조회, 차단 또는 삭제할 수 있어야 한다.
- FR-ADMIN-04: 관리자는 전체 신고 내역을 조회할 수 있어야 한다.
- FR-ADMIN-05: 관리자는 부적절한 채팅 메시지를 삭제할 수 있어야 한다.
- FR-ADMIN-06: 관리자는 전체 송금 내역을 조회할 수 있어야 한다.
- FR-ADMIN-07: 일반 사용자는 관리자 기능에 접근할 수 없어야 한다.

## 4. 보안 요구사항

### 4.1 인증과 비밀번호

- SR-AUTH-01: 비밀번호를 평문으로 저장하지 않는다.
- SR-AUTH-02: 안전한 단방향 비밀번호 해시를 사용한다.
- SR-AUTH-03: 로그인 성공 후 세션을 새로 생성한다.
- SR-AUTH-04: 로그인 실패 메시지로 아이디 존재 여부가 드러나지 않게 한다.
- SR-AUTH-05: 반복적인 로그인 시도를 제한한다.
- SR-AUTH-06: 비밀번호 변경 전에 현재 비밀번호를 다시 확인한다.

### 4.2 접근 제어

- SR-ACCESS-01: 인증이 필요한 기능은 서버에서 로그인 상태를 검사한다.
- SR-ACCESS-02: 상품 수정과 삭제 시 상품 소유자를 검사한다.
- SR-ACCESS-03: 관리자 기능은 서버에서 관리자 권한을 검사한다.
- SR-ACCESS-04: URL의 ID 값을 변경해 다른 사용자의 데이터에 접근할 수 없어야 한다.
- SR-ACCESS-05: 1:1 채팅은 대화 참여자만 조회할 수 있어야 한다.

### 4.3 입력값 검증

- SR-INPUT-01: 모든 입력값은 서버에서 다시 검증한다.
- SR-INPUT-02: 문자열 입력값에는 최대 길이를 적용한다.
- SR-INPUT-03: 상품 가격과 송금 금액은 정수 범위를 검사한다.
- SR-INPUT-04: SQL 쿼리는 ORM 또는 매개변수 바인딩을 사용한다.
- SR-INPUT-05: 사용자 출력값을 이스케이프하여 XSS를 방어한다.
- SR-INPUT-06: 상태 변경 요청에는 CSRF 토큰을 사용한다.

### 4.4 파일 업로드

- SR-FILE-01: 허용된 이미지 확장자만 업로드할 수 있어야 한다.
- SR-FILE-02: 업로드 파일의 크기를 제한한다.
- SR-FILE-03: 파일 확장자뿐 아니라 실제 이미지 형식을 확인한다.
- SR-FILE-04: 업로드 파일명은 서버에서 임의로 다시 생성한다.
- SR-FILE-05: 업로드 파일을 실행 가능한 소스 코드 위치에 저장하지 않는다.

### 4.5 오류 및 비밀정보

- SR-ERROR-01: 사용자 화면에 스택 트레이스나 내부 오류를 노출하지 않는다.
- SR-ERROR-02: 비밀키와 관리자 비밀번호는 환경변수로 관리한다.
- SR-ERROR-03: `.env`와 실제 데이터베이스를 GitHub에 올리지 않는다.
- SR-ERROR-04: 로그에 비밀번호, 세션 또는 민감정보를 기록하지 않는다.

### 4.6 송금 무결성

- SR-TRANSFER-01: 출금, 입금 및 송금 기록은 하나의 트랜잭션으로 처리한다.
- SR-TRANSFER-02: 송금 직전에 서버에서 잔액을 다시 검사한다.
- SR-TRANSFER-03: 클라이언트에서 전달된 잔액 값을 신뢰하지 않는다.
- SR-TRANSFER-04: 송금 실패 시 모든 데이터 변경을 취소한다.

## 5. 비기능 요구사항

- NFR-01: 모바일과 데스크톱에서 사용할 수 있는 반응형 화면을 제공한다.
- NFR-02: 사용자가 이해할 수 있는 오류 메시지를 제공한다.
- NFR-03: 프로그램 설치 및 실행 방법을 README에 작성한다.
- NFR-04: 데이터베이스 생성 과정을 자동화한다.
- NFR-05: 주요 기능과 보안 기능에 대한 테스트를 작성한다.
- NFR-06: 개발 및 유지보수가 쉽도록 기능별 Blueprint 구조를 사용한다.
- NFR-07: Python 의존성 버전을 `requirements.txt`에 기록한다.

## 6. 요구사항 변경 사항

과제에서는 사용자 간 송금을 요구하지만 실제 금융 서비스 연동은
교육용 프로젝트의 범위와 안전성을 넘어선다.

따라서 본 프로젝트에서는 회원에게 지급된 가상 잔액을 사용자 간에
이체하는 방식으로 요구사항을 구체화한다.

출금과 입금은 데이터베이스 트랜잭션을 사용하여 데이터의 일관성을 보장한다.

## 7. 완료 기준

다음 조건을 모두 만족하면 프로젝트가 완료된 것으로 판단한다.

- 일반 사용자와 관리자의 권한이 분리되어 있다.

- 완성된 프로그램이 GitHub 공개 저장소에 게시되어 있다.
